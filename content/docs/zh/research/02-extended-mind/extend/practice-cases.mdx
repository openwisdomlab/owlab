# 实践案例库

**Practice Cases: Maker Education Examples for the Three Integration Pathways**

---

## 使用说明

本案例库按认知整合的三条路径组织：

- **身体整合案例**：强调手势、动作、多感官体验
- **工具整合案例**：强调工具熟练度、AI辅助、递减支持
- **符号整合案例**：强调书写、图解、代码、原型

每个案例包含：
- 项目描述
- 三路径分析（哪条路径是重点？）
- 关键设计要点
- 常见陷阱与解决方案

## 一、身体整合案例

### 案例1.1：电路感知工作坊

**项目描述**：学生设计并制作一个LED呼吸灯电路

**目标年龄**：10-14岁

**时长**：3小时

#### 活动流程

```
阶段1: 身体预测 (30分钟)
├── 用手势比划电流流动方向
├── 预测"如果断开这里会怎样？"
└── 手持电线感受温度变化

阶段2: 触觉探索 (60分钟)
├── 在面包板上插接元件
├── 感受LED正负极的区别
├── 用手指感知电阻的阻力差异
└── 观察LED亮度变化（电压变化的视觉反馈）

阶段3: 预测-验证循环 (60分钟)
├── 预测："加大电阻会怎样？"
├── 动手调整
├── 观察结果
└── 更新理解

阶段4: 身体演示 (30分钟)
└── 用手势向同伴解释电路原理
```

#### 三路径分析

| 路径 | 比重 | 具体体现 |
|-----|------|---------|
| **身体整合** | ★★★ | 手势预测、触觉反馈、温度感知 |
| 工具整合 | ★★ | 使用面包板、万用表 |
| 符号整合 | ★ | 电路图记录（次要） |

#### 关键设计要点

<Callout type="info">
**为什么有效？**

1. **手势映射概念**：用手指沿电路比划，直接映射"电流流动"的抽象概念
2. **即时感知反馈**：LED亮度变化提供即时视觉反馈，温度变化提供触觉反馈
3. **预测-验证紧密**：每次调整前先预测，调整后立即观察
</Callout>

#### 常见陷阱

| 陷阱 | 问题 | 解决方案 |
|-----|------|---------|
| 跳过预测 | 直接操作，不激活预测网络 | 每次改动前必须口头预测 |
| 触觉缺失 | 使用虚拟仿真代替实物 | 始终从实物开始 |
| 解释过早 | 在学生探索前就讲原理 | 先让学生发现问题再解释 |

### 案例1.2：力与运动探索

**项目描述**：用身体动作探索牛顿三定律

**目标年龄**：12-16岁

**时长**：2小时

#### 活动流程

```
牛顿第一定律（惯性）
├── 活动：推动滑板车后松手
├── 身体感受：滑板车继续前进
└── 手势记录：用手势演示"保持运动"

牛顿第二定律（F=ma）
├── 活动：推不同重量的箱子
├── 身体感受：重的箱子更难推
└── 手势演示：大力→快加速，小力→慢加速

牛顿第三定律（作用与反作用）
├── 活动：两人站在滑板上互推
├── 身体感受：两人都向后移动
└── 手势演示：用双手相互推来解释
```

#### 关键设计要点

<Callout type="info">
**具身设计原则**

1. **全身参与**：不是用手指操作模型，而是用整个身体体验力
2. **概念-动作对应**：推力的大小直接映射为身体的用力感
3. **手势作为记忆锚点**：用手势"锁定"物理概念
</Callout>

### 案例1.3：编程的具身入门

**项目描述**：用身体动作学习编程基本概念

**目标年龄**：8-12岁

**时长**：90分钟

#### 活动设计

```
概念1: 顺序执行
├── 活动："机器人"游戏
├── 一人当"机器人"，接受指令（前进、转左、转右）
├── "程序员"必须按顺序给出指令
└── 身体体验：指令顺序错了，"机器人"会走错路

概念2: 循环
├── 活动：重复动作舞蹈
├── "重复3次：向前走两步，转圈"
├── 用身体执行循环
└── 手势：用画圈手势表示"重复"

概念3: 条件判断
├── 活动：有条件的动作游戏
├── "如果听到拍手，就跳；否则，原地不动"
├── 身体体验分支逻辑
└── 手势：用叉路手势表示"如果-否则"
```

#### 从具身到代码的桥梁

```
身体活动               →        代码表示
─────────────────────────────────────────────
"向前走两步"           →        forward(2)
"重复3次..."          →        for i in range(3):
"如果听到拍手"         →        if hear_clap():
```

## 二、工具整合案例

### 案例2.1：AI辅助项目设计

**项目描述**：设计一个自动浇花系统，AI作为脚手架

**目标年龄**：14-18岁

**时长**：6小时（分2次）

#### 递减支持路径设计

```
Level 3: 完整AI辅助 (新手)
├── AI提供完整传感器选型建议
├── AI解释每个组件的功能
└── AI生成参考代码框架

Level 2: 部分提示 (进阶)
├── AI仅在请求时提供建议
├── 学生自己查找组件规格
└── AI检查代码但不直接修改

Level 1: 仅验证功能 (熟练)
├── 学生独立完成设计
├── AI仅回答具体技术问题
└── 学生自己调试

Level 0: 独立完成 (掌握)
├── 无AI辅助
├── 学生独立完成类似项目
└── 作为能力检验
```

#### 三路径分析

| 路径 | 比重 | 具体体现 |
|-----|------|---------|
| 身体整合 | ★★ | 组装电路、测试传感器 |
| **工具整合** | ★★★ | AI递减支持、工具熟练度 |
| 符号整合 | ★★ | 电路图、代码、文档 |

#### AI使用检查点

<Callout type="warn">
**每次使用AI前问自己**

- [ ] 我自己先尝试过吗？
- [ ] 我是在寻求理解还是寻求答案？
- [ ] 我能解释AI的输出吗？
- [ ] 如果没有AI，我能完成核心部分吗？
</Callout>

#### 常见陷阱

| 陷阱 | 问题 | 解决方案 |
|-----|------|---------|
| AI替代思考 | 直接复制AI生成的代码 | 要求学生逐行解释代码 |
| 跳过Level | 急于完成，跳过独立尝试 | 设定每个Level的最短时间 |
| 无检验 | 完成后不知道自己学到什么 | 设置无AI的能力检验环节 |

### 案例2.2：3D打印工具熟练度

**项目描述**：从新手到熟练使用3D打印机

**目标年龄**：12-18岁

**时长**：渐进式（4周）

#### 工具整合阶梯

```
第1周: 观察与辅助
├── 观察导师操作
├── 辅助装卸材料
├── 身体记忆：丝料的手感、温度警示
└── 目标：理解基本流程

第2周: 指导下操作
├── 在导师监督下完成完整打印
├── 学习切片软件基础设置
├── 开始形成"手感"
└── 目标：能完成标准流程

第3周: 独立操作
├── 独立完成打印任务
├── 自己处理常见问题
├── 工具开始"透明"
└── 目标：流畅完成任务

第4周: 问题解决
├── 处理复杂问题（堵头、翘边）
├── 调整高级参数
├── 工具成为身体延伸
└── 目标：直觉判断问题
```

#### 工具整合检验标准

| 阶段 | 检验标准 |
|-----|---------|
| 观察期 | 能描述打印流程 |
| 指导期 | 能在提示下完成打印 |
| 独立期 | 能独立完成标准打印 |
| 熟练期 | 能诊断并解决问题，不需要思考基本操作 |

### 案例2.3：测量仪器整合

**项目描述**：万用表成为"看见电路"的眼睛

**目标年龄**：12-16岁

**时长**：3小时

#### 活动设计

```
阶段1: 预测与验证
├── 先用手势预测电路中各点的电压
├── 用万用表测量验证
├── 比较预测与实际
└── 更新心智模型

阶段2: 工具透明化练习
├── 重复测量不同电路
├── 减少看表盘的时间
├── 培养"表笔到位→读数"的自动反应
└── 目标：测量变成"看"的延伸

阶段3: 诊断应用
├── 给一个"故障"电路
├── 用万用表系统诊断
├── 万用表成为"找问题"的工具
└── 目标：工具整合完成
```

#### 整合检验

**问题**：学生能否在不看表盘的情况下，凭"手感"判断表笔是否接触良好？

**标志**：当学生开始"通过万用表看电路"而非"用万用表测电路"时，整合完成。

## 三、符号整合案例

### 案例3.1：设计日志工作坊

**项目描述**：通过书写推动项目思考

**目标年龄**：12-18岁

**时长**：贯穿项目全程

#### 日志结构设计

```
每次工作后的日志 (10-15分钟)

┌─────────────────────────────────────────────┐
│ 日期：______ 工作时长：______               │
├─────────────────────────────────────────────┤
│                                             │
│ 【今天做了什么？】                          │
│ （简要记录，3-5句）                         │
│                                             │
├─────────────────────────────────────────────┤
│                                             │
│ 【遇到什么问题？】                          │
│ （具体描述问题，而非只写"有问题"）          │
│                                             │
├─────────────────────────────────────────────┤
│                                             │
│ 【我的假设是什么？】                        │
│ （为什么出问题？我猜是因为...）             │
│                                             │
├─────────────────────────────────────────────┤
│                                             │
│ 【下一步计划】                              │
│ （明天要尝试什么？）                        │
│                                             │
├─────────────────────────────────────────────┤
│                                             │
│ 【草图/示意图】                             │
│ （画出关键部分，不需要漂亮）                │
│                                             │
└─────────────────────────────────────────────┘
```

#### 三路径分析

| 路径 | 比重 | 具体体现 |
|-----|------|---------|
| 身体整合 | ★ | 手写的具身性 |
| 工具整合 | ★ | 笔作为思考工具 |
| **符号整合** | ★★★ | 书写外化思维、草图可视化 |

#### 书写的认知功能

<Callout type="info">
**为什么写下来有用？**

1. **外化迫使具体化**："有问题"必须变成"LED不亮，可能是..."
2. **自我对话**：写完后重读，发现逻辑漏洞
3. **时间轴可见**：项目进展可追溯
4. **假设显性化**：预测被记录，可以后续验证
</Callout>

### 案例3.2：草图先行设计法

**项目描述**：用草图驱动产品设计

**目标年龄**：14-18岁

**时长**：4小时

#### 活动流程

```
阶段1: 快速草图 (30分钟)
├── 主题：设计一个手机支架
├── 规则：5分钟内画5个不同方案
├── 要求：不需要精美，只需要能传达想法
└── 产出：25-30个粗略草图

阶段2: 草图评审 (30分钟)
├── 自己审视所有草图
├── 标记：哪个最有潜力？为什么？
├── 发现：草图中有什么新想法是画的时候才想到的？
└── 选择：选3个进入下一轮

阶段3: 细化草图 (60分钟)
├── 对3个方案画更详细的草图
├── 添加尺寸标注
├── 考虑材料和制作方法
└── 产出：3个详细方案草图

阶段4: 快速原型 (120分钟)
├── 选1个方案制作实物原型
├── 用纸板/纸张快速制作
├── 测试原型（能放手机吗？稳吗？）
└── 产出：可测试的原型

阶段5: 反思 (30分钟)
├── 对比草图与原型
├── 草图中什么在原型中不可行？
├── 原型揭示了什么草图没想到的问题？
└── 更新草图，准备下一轮迭代
```

#### 草图的认知功能

| 阶段 | 草图的作用 |
|-----|-----------|
| 快速草图 | **生成想法**——画的过程中产生新想法 |
| 草图评审 | **发现可能**——在模糊的线条中看到新结构 |
| 细化草图 | **澄清意图**——迫使决定具体参数 |
| 原型后更新 | **整合学习**——将物理发现记录回符号层面 |

### 案例3.3：代码作为思考媒介

**项目描述**：通过编程理解算法思维

**目标年龄**：14-18岁

**时长**：3小时

#### 活动设计：排序算法

```
阶段1: 身体排序 (20分钟)
├── 5个学生拿着数字卡片
├── 不许说话，只能通过比较和交换位置来排序
├── 观察者记录交换了多少次
└── 身体体验：排序是"比较"和"交换"的组合

阶段2: 伪代码描述 (20分钟)
├── 用自然语言写下刚才的步骤
├── 发现模式：重复比较相邻两个，小的放前面
└── 产出：伪代码版本的冒泡排序

阶段3: 代码实现 (60分钟)
├── 将伪代码转为Python代码
├── 运行测试
├── 调试（如果有bug）
└── 代码作为"可执行的思想"

阶段4: 优化思考 (40分钟)
├── 问题：有没有更快的方法？
├── 回到身体：用不同策略排序
├── 发现：分而治之（快速排序的直觉）
└── 代码实现优化版本

阶段5: 可视化 (40分钟)
├── 让代码可视化排序过程
├── 看到算法"运行"
└── 符号（代码）→视觉（动画）→理解
```

#### 代码的符号整合功能

<Callout type="info">
**代码为什么是强大的思考工具？**

1. **精确性**："比较相邻两个"必须变成`if arr[i] > arr[i+1]`
2. **可执行**：想法可以被"运行"，立即看到结果
3. **可调试**：错误暴露思维漏洞，bug是学习信号
4. **可迭代**：修改代码重新运行，快速验证新想法
</Callout>

## 四、三路径整合案例

### 案例4.1：完整项目：智能台灯

**项目描述**：设计制作一个能感应环境亮度自动调节的台灯

**目标年龄**：14-18岁

**时长**：12小时（分4次）

#### 三路径整合设计

```
┌─────────────────────────────────────────────────────────────┐
│                    智能台灯项目                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  第1次：构思 (3小时)                                        │
│  ├─ [符号] 头脑风暴，画草图探索可能性                        │
│  ├─ [身体] 用手比划灯的形态，感受空间关系                   │
│  └─ [工具] AI辅助了解传感器选项（Level 3）                  │
│                                                             │
│  第2次：原型 (3小时)                                        │
│  ├─ [身体] 组装电路，触觉反馈                               │
│  ├─ [工具] 使用面包板和基础工具                             │
│  └─ [符号] 画电路图，记录连接方式                           │
│                                                             │
│  第3次：编程 (3小时)                                        │
│  ├─ [符号] 编写控制代码                                     │
│  ├─ [身体] 测试传感器响应（用手遮挡光线）                   │
│  └─ [工具] AI辅助调试（Level 2，仅在请求时介入）            │
│                                                             │
│  第4次：完善与展示 (3小时)                                  │
│  ├─ [身体] 制作外壳，手工组装                               │
│  ├─ [符号] 撰写项目报告，画最终示意图                       │
│  └─ [工具] 无AI辅助，独立完成（Level 0检验）                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 各阶段的三路径比重

| 阶段 | 身体整合 | 工具整合 | 符号整合 |
|-----|---------|---------|---------|
| 构思 | ★★ | ★ | ★★★ |
| 原型 | ★★★ | ★★ | ★ |
| 编程 | ★ | ★★ | ★★★ |
| 完善 | ★★ | ★ | ★★ |

#### 检验清单

**项目结束时检验**

- [ ] **身体整合**：学生能用手势解释电路工作原理吗？
- [ ] **工具整合**：学生能独立使用烙铁/万用表吗？移除AI后能完成类似任务吗？
- [ ] **符号整合**：学生的项目文档是否清晰？代码是否有注释？草图是否反映了设计演变？

## 案例设计原则总结

<Callout type="info">
**设计创客学习活动的三路径检查清单**

**身体整合**
- [ ] 是否有主动身体参与（不只是点击/观看）？
- [ ] 手势/动作是否映射核心概念？
- [ ] 是否提供即时多感官反馈？
- [ ] 是否设计了预测-行动-验证循环？

**工具整合**
- [ ] 工具是扩展能力还是替代能力？
- [ ] 是否设计了递减支持路径？
- [ ] 是否有无工具/无AI的能力检验？
- [ ] 工具使用是否趋向"透明"？

**符号整合**
- [ ] 是否鼓励外化表征（写、画、建模）？
- [ ] 草图是否被用作思考工具（而非只是记录）？
- [ ] 代码/文档是否被迭代改进？
- [ ] 是否有反思性书写环节？
</Callout>
