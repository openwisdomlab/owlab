# 评估框架

**Assessment Framework: Methods and Tools for Evaluating Cognitive Integration**

---

## 评估哲学

### 为什么传统评估不够？

传统评估聚焦于**最终产出**（作品质量、测试分数），但认知整合关注的是**过程性能力**：

| 传统评估 | 认知整合评估 |
|---------|-------------|
| 作品完成了吗？ | 学习者经历了什么认知过程？ |
| 答案正确吗？ | 学习者能独立解决问题吗？ |
| 代码能运行吗？ | 学习者理解代码为什么这样写吗？ |
| 项目成功了吗？ | 学习者从失败中学到了什么？ |

### 核心评估问题

```
三路径核心评估问题

身体整合：学习者是否通过身体行动建立了理解？
         → 检验：能否用手势/动作解释概念？

工具整合：工具是否成为了能力的延伸而非替代？
         → 检验：移除工具后能否独立完成核心任务？

符号整合：外部表征是否支持了思考过程？
         → 检验：笔记/草图是否反映了思维演变？
```

## 一、身体整合评估

### 1.1 手势表达评估

**评估场景**：要求学习者向他人解释项目中的核心概念

**评估标准**：

| 等级 | 描述 | 示例 |
|-----|------|-----|
| **4 - 自然整合** | 自发使用精准手势，手势与概念结构高度对应 | 解释电流时手指自然沿电路路径移动 |
| **3 - 有意识使用** | 在提示下使用手势，手势基本对应概念 | 提醒后能用手势演示力的方向 |
| **2 - 简单手势** | 使用一般性手势，与概念对应较弱 | 只是指点，没有动态演示 |
| **1 - 无手势** | 完全依赖语言，没有身体参与 | 只是口头描述 |

### 1.2 预测-验证能力评估

**评估场景**：在动手操作前，要求学习者预测结果

**评估方法**：

1. **预测记录**：在操作前记录学习者的预测
2. **实际观察**：记录实际结果
3. **反思质量**：评估学习者对预测误差的解释

**评估标准**：

| 等级 | 预测能力 | 反思能力 |
|-----|---------|---------|
| **4** | 预测准确或接近，能解释为什么 | 能清晰解释预测与实际的差异原因 |
| **3** | 预测有一定道理，部分正确 | 能识别差异，尝试解释 |
| **2** | 预测较随意，缺乏依据 | 能识别差异，难以解释 |
| **1** | 不愿预测或无法预测 | 无法反思差异 |

**评估记录表**：

```

│ 预测-验证能力评估                                           │

│ "我认为会发生：_______________________________________"     │
│ "因为：______________________________________________"      │

│ 【实际结果】                                                │

│ "我认为原因是：______________________________________"      │
│                                                            │

```

### 1.3 多感官参与观察

**评估场景**：观察学习者在动手活动中的感官参与

**观察指标**：

- [ ] **视觉参与**：观察细节、注意变化
- [ ] **触觉参与**：触摸材料、感受反馈
- [ ] **听觉参与**：注意声音变化（如电机声、材料声）
- [ ] **本体感受**：注意身体位置、力度控制

**评估标准**：

| 等级 | 描述 |
|-----|------|
| **4** | 多感官协同，主动利用不同感官获取信息 |
| **3** | 使用2-3种感官，有意识但不够自然 |
| **2** | 主要依赖视觉，偶尔使用其他感官 |
| **1** | 单一感官（通常是视觉），忽略其他感官信息 |

## 二、工具整合评估

### 2.1 工具透明性评估

**核心问题**：工具使用是否变得"透明"——不需要有意识地思考工具本身？

**评估方法**：观察学习者使用工具时的注意力分配

| 等级 | 描述 | 行为指标 |
|-----|------|---------|
| **4 - 透明** | 工具"消失"，注意力完全在任务上 | 流畅操作，不看工具本身 |
| **3 - 接近透明** | 大部分时间透明，偶尔需要注意工具 | 操作流畅，偶尔调整姿势/位置 |
| **2 - 半透明** | 需要分配注意力给工具操作 | 操作时需要看工具，有时停顿 |
| **1 - 不透明** | 工具操作本身消耗大量认知资源 | 操作生涩，频繁出错 |

**观察记录表**：

```
┌────────────────────────────────────────────────────────────┐
│ 工具透明性观察                                              │
├────────────────────────────────────────────────────────────┤
│ 工具：____________  任务：____________                      │
├────────────────────────────────────────────────────────────┤
│                                                            │
│ 注意力观察：                                                │
│ □ 注意力主要在任务目标                                     │
│ □ 注意力在工具和任务之间切换                               │
│ □ 注意力主要在工具操作本身                                 │
│                                                            │
│ 操作流畅度：                                                │
│ □ 流畅连续，无明显停顿                                     │
│ □ 大体流畅，偶有停顿                                       │
│ □ 频繁停顿，需要思考下一步                                 │
│ □ 操作生涩，频繁出错                                       │

### 2.2 独立能力检验

**核心问题**：移除工具/AI辅助后，学习者能否完成核心任务？

**检验方法**：

```
独立能力检验流程

1. 识别核心能力
   └── 项目中的核心技能是什么？（不是所有技能都需要独立）

2. 设计无工具任务
   └── 与项目类似但更简单的任务，要求独立完成

3. 执行检验
   └── 学习者独立完成，评估者观察

4. 能力评定
   └── 根据完成情况评定等级
```

**AI辅助项目的独立能力检验**：

| 等级 | 描述 | 检验方法 |
|-----|------|---------|
| **4 - 完全独立** | 能独立完成类似复杂度的任务 | 无AI完成类似项目 |
| **3 - 基本独立** | 能独立完成核心部分，细节可能需要查资料 | 无AI完成简化版任务 |
| **2 - 部分独立** | 能独立完成部分环节，关键环节需要帮助 | 分步骤检验各环节 |
| **1 - 依赖工具** | 移除AI后难以完成任务 | 观察到明显困难 |

## 三、符号整合评估

### 3.1 设计日志质量评估

**评估维度**：

| 维度 | 4分 | 3分 | 2分 | 1分 |
|-----|-----|-----|-----|-----|
| **问题描述** | 具体、可操作 | 清晰但略泛 | 模糊 | 缺失 |
| **假设提出** | 有理有据 | 有假设但依据不足 | 仅描述现象 | 无假设 |
| **反思深度** | 连接多个概念 | 有反思 | 表面反思 | 无反思 |
| **草图使用** | 草图驱动思考 | 草图辅助说明 | 简单图示 | 无草图 |
| **迭代可见** | 多版本演变清晰 | 有修改痕迹 | 单一版本 | 无记录 |

### 3.2 草图思考评估

**评估场景**：观察草图在设计过程中的作用

**评估标准**：

| 等级 | 描述 | 指标 |
|-----|------|-----|
| **4** | 草图驱动思考——在画的过程中产生新想法 | 多版本，明显演变，有"啊哈"时刻 |
| **3** | 草图辅助思考——帮助澄清已有想法 | 有修改，帮助决策 |
| **2** | 草图记录想法——想好后画出来 | 单一版本，主要用于展示 |
| **1** | 不使用草图或草图无意义 | 缺失或与设计无关 |

**评估问题**：
- 这个草图是画完后才有的想法，还是画之前就想好了？
- 草图有几个版本？版本之间有什么变化？
- 学习者能指出草图中的哪些部分帮助了思考？

### 3.3 代码理解评估

**评估场景**：检验学习者是否理解自己写的（或AI辅助写的）代码

**评估方法**：

1. **代码讲解**：要求学习者逐行解释代码
2. **修改挑战**：给一个修改要求，观察学习者如何修改
3. **错误定位**：故意引入bug，观察学习者如何调试

**评估标准**：

| 等级 | 代码讲解 | 修改能力 | 调试能力 |
|-----|---------|---------|---------|
| **4** | 能解释每行为什么这样写 | 能快速正确修改 | 能系统定位并修复 |
| **3** | 能解释大部分代码 | 能修改但需要尝试 | 能定位问题，修复需时间 |
| **2** | 能描述代码做什么，不清楚为什么 | 修改困难，多次尝试 | 难以定位问题 |
| **1** | 无法解释代码 | 无法独立修改 | 无法调试 |

### 4.2 学习者自评问卷

**用途**：让学习者反思自己的认知整合过程

```

## 五、评估实施建议

### 5.1 评估时机

| 时机 | 评估类型 | 重点 |
|-----|---------|-----|
| **项目前** | 前测/基线 | 了解起点水平 |
| **过程中** | 形成性评估 | 及时调整支持 |
| **项目后** | 总结性评估 | 评估整合效果 |
| **延迟** | 迁移检验 | 检验独立能力 |

### 5.2 评估者角色

<Callout type="info">
**评估的目的不是打分，而是促进学习**

- 评估结果应该**反馈给学习者**，帮助其自我改进
- 评估应该**尊重过程**，而非只看结果
- 评估应该**识别优势**，而非只找问题
</Callout>

### 5.3 与M09评估模块的关系

本评估框架聚焦于**认知整合过程**，补充M09模块的**项目成果评估**：

| 评估维度 | 本框架 (L02) | M09 评估 |
|---------|-------------|---------|
| 评估对象 | 认知过程、整合度 | 项目成果、技能展示 |
| 核心问题 | "学习者如何思考？" | "学习者做出了什么？" |
| 证据类型 | 观察、反思、过程记录 | 作品、演示、作品集 |

**建议**：两种评估结合使用，既关注成果质量，也关注学习过程。
