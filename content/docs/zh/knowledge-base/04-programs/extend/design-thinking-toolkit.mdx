# 设计思维与未来思维工具库

## 概述

设计思维 (Design Thinking) 已成为21世纪解决复杂问题的核心方法论。它不仅是设计师的专属工具，更是一种人人可学、可用的思维方式。本文档整合经典设计思维方法与新兴的未来导向设计方法，为 OWL 提供系统的设计教育工具库。

**设计思维的价值**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    设计思维的核心价值                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   【以人为本】                     【拥抱不确定性】                  │
│   从真实用户需求出发                不害怕模糊和未知                 │
│   深入理解用户情境                  在迭代中逐步清晰                 │
│                                                                      │
│   【整合思维】                     【行动导向】                      │
│   既分析又综合                      快速原型而非完美计划              │
│   连接看似无关的元素                从做中学、做中改                 │
│                                                                      │
│   【乐观主义】                     【协作创新】                      │
│   相信更好的解决方案存在            多元团队共创                     │
│   失败是通向成功的必经之路          突破专业边界                     │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## Part I: 经典设计思维

## 1. 设计思维五阶段

### 1.1 概览

**Stanford d.school 设计思维模型** [E2]:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    设计思维五阶段                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│   │ 共情    │   │ 定义    │   │ 构思    │   │ 原型    │   │ 测试    │
│   │ Empathize│  │ Define  │   │ Ideate  │   │ Prototype│  │ Test    │
│   │         │   │         │   │         │   │         │   │         │
│   │ 理解    │ → │ 聚焦    │ → │ 发散    │ → │ 具象化  │ → │ 验证    │

│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 阶段详解

#### 阶段1: 共情 (Empathize)

**目标**: 深入理解用户的需求、行为、动机和痛点

**核心方法**:

| 方法 | 描述 | 工具 | 适用场景 |
|------|------|------|----------|
| **用户访谈** | 深度对话了解用户 | 访谈提纲、录音 | 了解动机和情感 |
| **观察法** | 在真实情境中观察 | 观察记录表 | 发现隐性需求 |
| **沉浸体验** | 亲身体验用户处境 | 角色扮演 | 建立同理心 |
| **影子跟踪** | 跟随用户一整天 | 记录工具 | 理解完整旅程 |
| **极端用户** | 访谈边缘用户 | 访谈提纲 | 发现创新机会 |

**共情地图模板**:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    共情地图 (Empathy Map)                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│        【想法与感受】                    【看到什么】                │
│        ─────────────                    ─────────────               │
│        内心想什么？                      环境中看到什么？            │
│        有什么感受？                      朋友在做什么？              │
│        什么让TA担心？                    市场提供什么？              │
│                                                                      │
│              ┌─────────────────────────┐                            │
│              │         用户           │                            │
│              └─────────────────────────┘                            │
│                                                                      │
│        【说什么与做什么】                【听到什么】                │
│        ───────────────────              ─────────────               │
│        可能说什么？                      朋友说什么？                │
│        观察到的行为？                    同事说什么？                │
│        对他人的态度？                    媒体传达什么？              │
│                                                                      │
│   ─────────────────────────────────────────────────────────────────  │
│        【痛点 Pain】                    【收获 Gain】                │
│        ─────────────                    ─────────────               │
│        害怕什么？                        想要达成什么？              │
│        什么让TA沮丧？                    如何衡量成功？              │
│        有什么障碍？                      什么策略可能有效？          │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

#### 阶段2: 定义 (Define)

**目标**: 将共情阶段的洞察综合为清晰的问题陈述

**核心产出**: POV（观点陈述）和 HMW（我们如何能...）

**POV 公式**:

```
【用户】______________________
【需要】______________________
【因为/洞察】______________________

示例：
【用户】社区独居老人
【需要】一种简单可靠的方式联系家人和获得帮助
【因为】现有技术产品对他们来说操作复杂，且担心成为负担
```

**HMW（How Might We）问题转化**:

| 原问题 | HMW 转化 |
|--------|----------|
| 老人不会用智能手机 | HMW 让老人无需学习就能联系家人？ |
| 老人担心成为负担 | HMW 让寻求帮助变得自然而非求助？ |
| 老人孤独 | HMW 创造自然的社交连接机会？ |

#### 阶段3: 构思 (Ideate)

**目标**: 生成尽可能多的创意解决方案

**核心原则**:
- **数量优先**: 追求数量，后再筛选质量
- **延迟判断**: 不过早否定想法
- **鼓励狂野**: 疯狂的想法可能有可行的内核
- **组合叠加**: 在他人想法上构建

**头脑风暴规则**:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    头脑风暴规则                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   ✓ 延迟判断 ─── 不说"不行"、"但是"                                │
│   ✓ 鼓励狂野 ─── 疯狂的想法欢迎                                     │
│   ✓ 数量优先 ─── 追求数量，目标100+                                 │
│   ✓ 视觉化   ─── 画出来而非只说                                     │
│   ✓ 一个对话 ─── 一次一人发言                                       │
│   ✓ 在他人想法上构建 ─── "是的，而且..."                           │
│   ✓ 聚焦话题 ─── 围绕 HMW 问题                                      │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

**构思方法矩阵**:

| 方法 | 描述 | 何时使用 |
|------|------|----------|
| **经典头脑风暴** | 自由联想生成想法 | 初期发散 |
| **逆向思维** | "如何让问题更糟" | 打破思维定势 |
| **类比法** | 从其他领域借鉴 | 寻找新视角 |
| **SCAMPER** | 系统性创新问题 | 改进现有方案 |
| **随机输入** | 随机词汇激发联想 | 突破瓶颈 |
| **最坏想法** | 故意想坏主意再反转 | 突破心理障碍 |

**SCAMPER 创新检查表**:

| 字母 | 问题 | 示例 |
|------|------|------|
| **S** Substitute | 可以替换什么？ | 用AI替换人工客服 |
| **C** Combine | 可以组合什么？ | 手机+相机+电脑 |
| **A** Adapt | 可以适配什么？ | 将游戏机制适配到学习 |
| **M** Modify | 可以放大/缩小什么？ | 缩小电脑成手表 |
| **P** Put to other use | 可以另作他用？ | 轮胎做秋千 |
| **E** Eliminate | 可以消除什么？ | 无线充电消除线缆 |
| **R** Rearrange/Reverse | 可以重排/反转？ | 先用后付 |

#### 阶段4: 原型 (Prototype)

**目标**: 快速将想法具象化以便测试和交流

**原型的目的**:
- 思考的工具而非最终产品
- 与用户沟通的媒介
- 快速验证假设
- 低成本试错

**原型保真度光谱**:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    原型保真度光谱                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │

│                                                                      │
│   ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐      │
│   │ 草图   │  │ 纸模型 │  │ 线框图 │  │ 可交互 │  │ 功能性 │      │
│   │        │  │        │  │        │  │ 原型   │  │ 原型   │      │
│   └────────┘  └────────┘  └────────┘  └────────┘  └────────┘      │
│                                                                      │
│   成本: 低 ─────────────────────────────────────────→ 高            │
│   速度: 快 ─────────────────────────────────────────→ 慢            │
│   可修改性: 高 ─────────────────────────────────────→ 低            │
│   用户理解: 需想象 ─────────────────────────────────→ 直观          │
│                                                                      │
│   建议: 尽可能从低保真开始，验证后再逐步提高保真度                  │

```

**原型方法**:

| 方法 | 描述 | 适用类型 |
|------|------|----------|
| **纸原型** | 纸和笔快速绘制 | 界面、流程 |
| **角色扮演** | 人扮演产品/服务 | 服务设计 |
| **故事板** | 分镜展示场景 | 体验流程 |
| **乐高/纸板** | 快速搭建物理模型 | 产品形态 |
| **视频原型** | 拍摄使用场景 | 复杂交互 |
| **绿野仙踪** | 人在幕后模拟系统 | 智能系统 |

#### 阶段5: 测试 (Test)

**目标**: 从真实用户获取反馈，验证和改进方案

**测试原则**:
- 带着开放心态而非证明自己正确
- 观察行为而非只听意见
- 问"为什么"深挖
- 测试越早越好，失败成本越低

## Part II: 未来导向的设计方法

## 2. 推测性设计 (Speculative Design)

### 2.1 什么是推测性设计

**核心定义** [E2]:
推测性设计 (Speculative Design) 是一种利用设计来探索和批判可能的未来的方法。它不以解决当前问题为目标，而是通过创造未来场景来引发讨论和反思。

**推测性设计 vs 传统设计**:

| 维度 | 传统设计 | 推测性设计 |
|------|----------|------------|
| **目标** | 解决当前问题 | 引发思考和讨论 |
| **时间** | 现在 | 可能的未来 |
| **受众** | 用户/消费者 | 公众/社会 |
| **成功标准** | 市场成功 | 引发有意义的辩论 |
| **产出** | 可用产品 | 概念产品/场景 |
| **立场** | 中立/服务用户 | 批判性/挑战假设 |

### 2.2 未来锥理论

**Stuart Candy 的未来锥** [E2]:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    未来锥 (Futures Cone)                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│                              现在                                    │
│                               │                                      │
│                              ╱│╲                                    │
│                             ╱ │ ╲    可能的 (Possible)              │
│                            ╱  │  ╲   物理定律允许的一切              │
│                           ╱   │   ╲                                  │
│                          ╱ ╱──│──╲ ╲  似真的 (Plausible)            │
│                         ╱ ╱   │   ╲ ╲ 基于当前知识可能发生的        │
│                        ╱ ╱ ╱──│──╲ ╲ ╲                              │
│                       ╱ ╱ ╱   │   ╲ ╲ ╲ 很可能的 (Probable)        │
│                      ╱ ╱ ╱    │    ╲ ╲ ╲ 当前趋势延续的结果        │
│                     ╱ ╱ ╱  ╱──│──╲  ╲ ╲ ╲                           │
│                    ╱ ╱ ╱  ╱   │   ╲  ╲ ╲ ╲ 偏好的 (Preferable)     │

└─────────────────────────────────────────────────────────────────────┘
```

### 2.3 推测性设计方法

**设计虚构 (Design Fiction)**:

| 步骤 | 活动 | 产出 |
|------|------|------|
| **1. 选择议题** | 确定要探讨的社会/技术议题 | 议题陈述 |
| **2. 趋势研究** | 研究相关趋势和信号 | 趋势地图 |
| **3. 场景构建** | 构建可能的未来场景 | 场景叙事 |
| **4. 设计日常物品** | 设计该未来中的日常物品 | 概念产品 |
| **5. 呈现与讨论** | 展示并引发讨论 | 讨论记录 |

### 2.4 教育中的推测性设计

**推测性设计培养的能力**:

| 能力 | 说明 | 培养方式 |
|------|------|----------|
| **未来想象力** | 想象不同于当下的可能 | 场景构建练习 |
| **批判性思维** | 质疑"理所当然" | 分析隐含假设 |
| **价值观反思** | 思考什么是值得追求的 | 偏好未来讨论 |
| **系统思维** | 理解变化的连锁反应 | 后果推演 |
| **伦理敏感性** | 考虑技术的社会影响 | 利益相关者分析 |

**适合教育的推测性设计主题**:

| 主题 | 可能探讨的问题 | 适合年龄 |
|------|----------------|----------|
| **AI与工作** | 如果AI能做大部分工作，人类做什么？ | 12+ |
| **隐私与监控** | 如果一切都被记录，生活会怎样？ | 10+ |
| **气候适应** | 如果海平面上升2米，城市会怎样？ | 10+ |
| **基因编辑** | 如果可以定制婴儿，社会会怎样？ | 14+ |
| **虚拟现实** | 如果虚拟世界比现实更"好"，会怎样？ | 12+ |
| **长寿社会** | 如果人人活到150岁，社会会怎样？ | 12+ |

## 3. 批判性设计 (Critical Design)

### 3.1 什么是批判性设计

**核心定义** [E2]:
批判性设计通过设计来质疑主流价值观、挑战假设、引发讨论，而非解决问题或服务商业。

**批判性设计的特征**:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    批判性设计 vs 肯定性设计                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   肯定性设计 (Affirmative)         批判性设计 (Critical)            │
│   ─────────────────────            ────────────────────             │
│                                                                      │
│   解决问题                          提出问题                         │
│   提供答案                          引发思考                         │
│   服务市场                          质疑市场                         │
│   接受现状                          挑战现状                         │
│   强调功能                          强调意义                         │
│   追求舒适                          制造不适                         │
│   确认期望                          颠覆期望                         │
│   取悦用户                          挑衅用户                         │
│   适应世界                          改变世界观                       │
│                                                                      │
│   两者都有价值，关键是有意识地选择                                  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 批判性设计方法

**质疑策略**:

| 策略 | 描述 | 示例 |
|------|------|------|
| **极端化** | 将趋势推向极端 | 如果社交媒体记录一切？ |
| **颠倒** | 反转常规假设 | 如果隐私成为奢侈品？ |
| **并置** | 将不相关事物放在一起 | 宠物+监控技术 |
| **讽刺** | 通过夸张揭示荒谬 | "完美"的量化自我 |
| **陌生化** | 让熟悉变得陌生 | 重新审视日常物品 |

### 3.3 教育中的批判性设计

**培养批判性思维的设计活动**:

| 活动 | 描述 | 讨论焦点 |
|------|------|----------|
| **重新设计日常物品** | 为不同价值观重新设计常见物品 | 物品隐含的价值观 |
| **反广告设计** | 设计揭示产品负面影响的广告 | 消费主义批判 |
| **替代历史设计** | 如果某技术未被发明，世界会怎样 | 技术并非必然 |
| **为边缘群体设计** | 为被忽视的群体设计 | 设计中的偏见 |
| **设计的设计** | 分析现有设计隐含的假设 | 解读设计语言 |

## 4. 参与式设计 (Participatory Design)

### 4.1 什么是参与式设计

**核心定义** [E2]:
参与式设计 (Participatory Design / Co-Design) 是一种将最终用户和利益相关者作为设计过程中的共同设计者而非仅仅信息来源的方法。

**参与阶梯**:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    参与程度阶梯                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   高 ─────────────────────────────────────────────────────── 高     │
│   参与程度                                           影响力          │
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ 共同决策 (Co-decision)                                      │   │
│   │ 用户与设计师共同做出设计决策                                │   │
│   └─────────────────────────────────────────────────────────────┘   │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ 共同创造 (Co-creation)                                      │   │
│   │ 用户参与创造设计方案                                        │   │
│   └─────────────────────────────────────────────────────────────┘   │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ 协商 (Consultation)                                         │   │
│   │ 设计师征求用户意见并考虑                                    │   │
│   └─────────────────────────────────────────────────────────────┘   │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │ 告知 (Information)                                          │   │
│   │ 设计师告知用户设计决定                                      │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│   低 ─────────────────────────────────────────────────────── 低     │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 共创工作坊方法

**常用共创工具**:

| 工具 | 描述 | 适用阶段 |
|------|------|----------|
| **文化探针** | 让参与者记录日常生活 | 前期研究 |
| **卡片分类** | 参与者组织信息结构 | 定义阶段 |
| **故事板共创** | 共同构建场景故事 | 构思阶段 |
| **乐高认真玩** | 用乐高建模想法 | 构思阶段 |
| **原型共建** | 共同制作原型 | 原型阶段 |
| **服务蓝图工作坊** | 共同绘制服务流程 | 系统设计 |

### 4.3 教育中的参与式设计

**让学生成为共同设计者**:

| 场景 | 参与式设计应用 | 教育价值 |
|------|----------------|----------|
| **学习空间设计** | 学生参与设计教室/创客空间 | 主人翁意识 |
| **课程设计** | 学生参与设计学习活动 | 学习动机 |
| **社区改善** | 学生参与社区问题解决 | 公民参与 |
| **产品开发** | 为真实用户共创产品 | 用户思维 |

## 5. 系统设计与转型设计

### 5.1 系统思维在设计中的应用

**系统思维设计原则**:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    系统设计思维原则                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   【关注关系而非组件】                                               │
│   系统的行为由组件间的关系决定，而非组件本身                        │
│                                                                      │
│   【理解反馈循环】                                                   │
│   识别正反馈（放大）和负反馈（平衡）循环                            │
│                                                                      │
│   【考虑涌现特性】                                                   │
│   整体具有部分不具备的特性                                          │
│                                                                      │
│   【识别杠杆点】                                                     │
│   小的改变可能引发大的系统变化                                      │
│                                                                      │
│   【考虑时间延迟】                                                   │
│   行动与结果之间可能有显著延迟                                      │
│                                                                      │
│   【接受不确定性】                                                   │
│   复杂系统不可完全预测                                              │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.2 系统图工具

**因果循环图 (Causal Loop Diagram)**:

```
示例: 社交媒体使用与心理健康

                    使用时间 ─────(+)────→ 社交比较
                       ↑                         │
                      (+)                       (+)
                       │                         ↓
                    无聊感 ←────(+)──── 负面情绪
                       ↑                         │
                      (+)                       (−)
                       │                         ↓
                    逃避 ←─────────────── 自我价值感

(+) 表示同向变化：一个增加，另一个也增加
(−) 表示反向变化：一个增加，另一个减少
循环方向: 正反馈循环会自我强化
```

### 5.3 转型设计 (Transition Design)

**转型设计关注长期、系统性的社会变革**:

| 层面 | 焦点 | 时间尺度 |
|------|------|----------|
| **产品/服务设计** | 解决即时问题 | 短期 |
| **系统设计** | 改变系统规则 | 中期 |
| **转型设计** | 改变深层价值观和范式 | 长期 |

## 7. 常见问题

<details>
<summary>设计思维和科学方法有什么区别？</summary>

**相似之处**:
- 都强调从问题出发
- 都需要验证假设
- 都强调迭代

**主要区别**:

| 维度 | 科学方法 | 设计思维 |
|------|----------|----------|
| 目标 | 发现真理 | 创造价值 |
| 问题类型 | 已存在的问题 | 需要被定义的问题 |
| 思维模式 | 主要分析性 | 分析+综合 |
| 对错误 | 需要避免 | 期待和学习 |
| 结果 | 唯一正确答案 | 多种可能方案 |

**互补使用**: 科学方法用于验证，设计思维用于创造。
</details>

<details>
<summary>推测性设计是"无用"的设计吗？</summary>

**不是"无用"，而是"不同的有用"**:

推测性设计的价值不在于产出可售卖的产品，而在于:
1. **引发公共讨论**: 让公众参与技术和社会议题的讨论
2. **影响决策**: 帮助政策制定者看到可能的未来
3. **塑造想象**: 扩展我们对可能性的想象
4. **培养批判性思维**: 质疑"理所当然"的假设
5. **激发行动**: 想象偏好的未来，激发改变的动力

教育价值: 培养学生成为技术和社会的批判性参与者，而非被动接受者。
</details>

<details>
<summary>如何在有限时间内运用设计思维？</summary>

**缩时版设计思维**:

| 时长 | 可行的活动 |
|------|-----------|
| **90分钟** | 快速共情→HMW→头脑风暴→纸原型 |
| **半天** | 简化的完整流程，每阶段30-45分钟 |
| **一天** | 完整流程，有时间深入每个阶段 |
| **多天** | 每阶段一天，真正深入研究用户 |

关键: 即使时间短，也要完成从共情到测试的完整循环，而非只做前面几步。
</details>
